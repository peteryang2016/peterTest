项目笔记

2.真实项目中并不是所有的组件都和REDUX
有关系，所以对于那些和REDUX没关系的组件，我们不要在connect高阶处理了（同理对于不需要使用路由中属性的一些组件，也没必要withRouter....）;也就是尽可能少使用高阶组件，因为高阶组件都是利用柯理化函数思想，形成闭包嵌套，这样导致很多栈内存不销毁，影响性能。   
3.真实项目中，完成数据绑定，我们可以按照以下方案处理：
    方案一：第一次加载组件之前或之后（willMount/Didmount），发送ajax请求，等待数据请求通过后，把请求回来的数据更新组件内部的状态信息（第一次加载的是空数据，第二次更新的时候加载真实的数据）从而重新渲染组件，展示真实的内容。
    弊端：在路由切换的时候，当前组件很有可能需要重新加载（组件完成了从页面移除到在展示的过程，这样需要从constructor从头加载组件），这样就导致，只要路由切换到这个组件，都要重新发送一次ajax请求，对于一些不是随时更新数据的组件，这样处理会增加HTTP请求的次数，增重服务器的处理压力！
    方案二：每一次加载组件，我们首先验证redux中是否存储了 展示信息，如果有，直接从redux获取即可，如果没有，发送一个dispatch派发，在派发的acition Creator中基于ajax获取数据，把获取的数据传递给reducer，把信息存储到redux中，redux中的信息更改，那么用到它的组件也会重新渲染
    弊端：某些特定的案例中会存在一些问题，需要额外处理，例如（在个人中心，A用于登录成功，我们进入个人信息页面，首先会把Ade信息存储到redux中，这样只要进入到信息页，展示都是A的信息，不会重新从服务器获取最新的信息，即使A的信息已经改变，或者是登录的用户已经变为B了，都不会改变）这种情， 我们需要在一些其他操作的时候，（例如：重新登录、修改用户信息、退出登录等操作），都才需要redux中存储的人心信息更新才可以！

4.在redux-promise中间件使用的时候，action-creator中返回的action对象，传递给reducer的action数据（从服务器获取的数据，开始返回的是一个promise）中的属性名必须是payload（严格区分大小写），只有这样，当promise成功，中间件才会帮助我们重新发送一次派发给reducer，然后把获取的数据信息跟新redux容器中的状态。如果强迫症不想使用payload作为属性名，那么可以用redux-Thunk来解决，使用redux-Thunk可以不用payload作为属性名。

5.实现购物车流程
    点击加入购物车
        1.向服务器发送请求，让服务器存储当前新家的信息（为了防止页面刷新，购物车信息消失：redux中的信息在页面刷新后没有了；可以跨平台（前提是用户已经登录）；）
        2.当服务器返回存储成功后，我们把信息往redux中存储一份（建议最好是：从服务器获取我的购物车信息，存储到redux中）
            2.1存储到redux中，目的是以后切换到我的购物车页面，没必要总是从服务器获取，从redux获取也可以，提高性能
            2.2之所以是从服务器获取信息存储到redux中，因为服务器的信息是最准确的，即使页面刷新了，即使某些操作我们忘记向redux中存储了，每一次派发都可以获取最新的信息（只要你向服务器发送了即可）
